# Punto de partida

Imagina que después de haber participado en un hackaton tenemos un MVP de una herramienta 
para gestión de bienes. Es una aplicación en la que tenemos un registro de los objetos 
que tenemos, otro para las personas que tienen asignado el objeto y un registro con los 
movimientos de los objetos y el historial de objetos de cada persona.

En otras palabras, un CRUD de objetos (en el código y de ahora en adelante *Assets*), otro de las personas a las que 
se les asignarán los objetos (*Holders*) y por último un historial de los movimientos 
de objetos entre personas (*Records*).

## ¿Qué tenemos que hacer?

### Arnés de seguridad
Después de esa hackaton quieres seguir con el proyecto y empezar a añadir nuevas funcionalidades.
Sin embargo, durante la hackaton tu equipo decidió no hacer tests de ningún tipo. 
Por tanto, primero deberás asegurarte de que no vas a romper nada creando un arnés 
de seguridad con [Cypress](https://docs.cypress.io/guides/references/best-practices.html).

### Añadir un filtro de búsqueda en cada vista
El MVP no está mal, pero pide a gritos un buscador con el que filtrar los Assets, 
Holders y Records. Para crear este nuevo componente vamos a seguir TDD creando los tests
antes que el propio componente. De esta manera diseñaremos el componente en base a las
especificaciones y las iremos validando a la vez que escribimos el código.
